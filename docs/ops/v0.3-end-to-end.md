# v0.3 End-to-End Verification Runbook

Audience: the user (host of `192.168.88.183`) doing the add-on install
themselves. The agent cannot run the add-on side of the loop (no HA
supervisor, no build context on the agent's host).

This runbook verifies the v0.3 changes on a real HA add-on install:
RAG-loaded orchestrator, soft card-type validation, feedback loop,
independent embedding endpoint. The hha-knowledge Linter (v0.3.2.2)
and learn.ts (v0.3.2.4) are pure hha-knowledge-side and don't need
the add-on — verify them via `pnpm scripts/lint` / `pnpm scripts/learn`
separately.

## Pre-flight

1. The v0.3 daemon code is committed + pushed to `master` (or a
   `v0.3.0` tag). CI will build + push `ghcr.io/lxgxdx/ha-ai-designer:{master,latest}`
   or `:{0.3.0}`.
2. The GHCR package has been set to **Public** (see CLAUDE.md 14 课 #5).
   If the image is still private, the install in step 2 will fail with
   a 401.
3. `hha-knowledge` exists at `E:\Claude\hha-knowledge\` with all 6 wiki
   articles (run `pnpm scripts/lint` once to confirm — expect
   `0 fail  3 warn  0 pass`).

## 1. Sync hha-knowledge to HA

The add-on's new `map: ["share:rw"]` bind-mounts the supervisor share
root at `/share` inside the container. The daemon will look for
`/share/hha-knowledge/wiki/index.md`. So you need that path to exist
on the HA host.

Easiest one-liner, from your dev machine, using SSH (CLAUDE.md 14 课 #8
notes that the host is `192.168.88.183`, user `hassio`, password the
same as the supervisor):

```bash
# from the machine that has E:\Claude\hha-knowledge\
scp -r E:\Claude\hha-knowledge hassio@192.168.88.183:/usr/share/hassio/share/
```

If you have passwordless sudo on the HA host:

```bash
ssh hassio@192.168.88.183 'sudo mkdir -p /usr/share/hassio/share && sudo chown hassio /usr/share/hassio/share'
scp -r E:\Claude\hha-knowledge hassio@192.168.88.183:/usr/share/hassio/share/
```

After this:

```bash
ssh hassio@192.168.88.183 'ls -la /usr/share/hassio/share/hha-knowledge/wiki/'
# expect: index.md  log.md  AGENTS.md  api/  cards/  dashboard-yaml/  entities/  jinja2/
```

To re-sync after wiki changes (e.g. after you run `pnpm scripts/learn`
and `cp .pending/cards/...md wiki/cards/...md`), just re-run the scp.
You don't need to restart the add-on — `initRagStore()` compares the
wiki's max mtime against `${DATA_DIR}/.rag.db`'s `last_indexed_mtime`
on every daemon start.

## 2. Upgrade the add-on

In HA → Settings → Add-ons → HA AI Designer → "Update" tab → Refresh.
The new `:master` (or `:0.3.0`) image should appear. Click Update.

If you've never set the visibility to Public, the install will pull
an image but the supervisor may refuse to start it. Check the
supervisor log for "denied" / "401" / "auth".

## 3. Configure knowledge_dir

Configuration tab → add the new field:

- `knowledge_dir`: `hha-knowledge` (the default, matches what you scp'd
  in step 1; if you put it under a different subdir of `/share/`,
  change this accordingly).

Other fields to confirm:
- `llm_provider`, `llm_base_url`, `llm_model`, `llm_api_key` — same
  as before. (v0.3.1 doesn't change them.)
- `embedding_base_url` / `embedding_model` / `embedding_api_key` —
  **v0.3.1.1, new optional fields.** For a first verification, leave
  them blank and rely on the chat LLM's `/v1/embeddings` endpoint.
  If your chat provider doesn't serve embeddings (most custom / Ollama
  setups), set these to your local `infinity` + `BAAI/bge-m3`
  endpoint.

## 4. Start the add-on

Start. Watch the add-on Log tab. You should see, in order:

```
[INFO] Persisting config to /data/config.json (apiKey masked in logs)
[INFO]   ha: supervisor token (baseUrl=http://supervisor/core)
[INFO]   llm: <provider> <model> (apiKey masked)
[INFO]   llm.embedding: <model>   (if you set one)
[INFO] Cleaning up any previous daemon/web processes...
[INFO] Starting daemon on port 7456...
[INFO] Starting web UI on port 3000...
```

Then on the daemon process itself (`/data/logs/daemon.log` via
`docker exec addon_<id> cat /data/logs/daemon.log`):

```
{"level":20,"time":...,"msg":"internal auth token loaded; web must present X-Addon-Internal-Token"}
{"level":30,"time":...,"msg":"ha-ai-designer daemon listening","port":7456}
```

## 5. Verify RAG init (the new v0.3.0 surface)

```bash
ssh hassio@192.168.88.183
docker exec addon_local_ha_ai_designer cat /data/logs/daemon.log | grep -E "wiki|RAG|embedding"
```

Expected (with embedding configured):

```
{"level":20,"msg":"wiki summaries loaded","knowledgeDir":"/share/hha-knowledge","bytes":N}
{"level":20,"msg":"RAG: embedding chunks","articles":6,"chunks":N,"dim":1536}
{"level":20,"msg":"RAG: reindex done","chunks":N,"dim":1536}
```

Expected (without embedding configured — RAG disabled, v0.3.0 mode only):

```
{"level":20,"msg":"RAG disabled: llm.embeddingModel not set in data/config.json"}
```

If you see neither, the most likely culprit is a wrong `HA_KNOWLEDGE_DIR`.
Check:

```bash
docker exec addon_local_ha_ai_designer ls -la /share/hha-knowledge/wiki/
docker exec addon_local_ha_ai_designer env | grep HA_KNOWLEDGE_DIR
```

## 6. Verify in the Web UI

Open the HA sidebar → "AI Designer" → `/chat`. Send a brief like:

> 做一个全屋概览，深蓝主题，灯为主，简洁。

You should see (in the LLM real-time output pane) a system prompt
section that starts with `## Available knowledge (from hha-knowledge
wiki)`, followed by the article list. If RAG is enabled, also a
`## Relevant knowledge (RAG top-k ...)` block.

After the YAML renders, you should see the new feedback row:
`这一稿对不对？ 👍 👎 加评论`

Click 👎 and add a comment. Then verify the JSONL line landed:

```bash
docker exec addon_local_ha_ai_designer cat /share/hha-knowledge/.feedback/feedback.jsonl
# expect: {"ts":"...","brief":"...","yaml":"...","rating":2,"comment":"...","entityRefs":[...]}
```

## 7. Run scripts/learn against the feedback (v0.3.2.4)

Back on your dev machine (Windows), with `LLM_BASE_URL` / `LLM_API_KEY`
/ `LLM_MODEL` exported (the same ones the add-on uses, or a separate
BYOK for the wiki-rewrite work):

```powershell
$env:LLM_BASE_URL = "https://api.minimaxi.com/v1"
$env:LLM_API_KEY = "sk-cp-..."
$env:LLM_MODEL = "MiniMax-Text-01"
$env:HA_KNOWLEDGE_DIR = "E:\Claude\hha-knowledge"

cd E:\Claude\hha-knowledge\scripts
pnpm learn --article wiki/cards/built-in-cards-overview.md
```

This will write a candidate rewrite to
`E:\Claude\hha-knowledge\.pending\cards\built-in-cards-overview.md`
and back the original up to
`.pending\archive\built-in-cards-overview.md.bak-<ts>`. It will NOT
modify the live wiki. The next step is yours:

```bash
# accept
cp E:\Claude\hha-knowledge\.pending\cards\built-in-cards-overview.md E:\Claude\hha-knowledge\wiki\cards\
# or reject
rm E:\Claude\hha-knowledge\.pending\cards\built-in-cards-overview.md
# then re-sync to HA and the daemon will re-index on next start
scp -r E:\Claude\hha-knowledge hassio@192.168.88.183:/usr/share/hassio/share/
```

## 8. Cleanup (when you're done verifying)

Re-sync the wiki one more time so any reject / accept decisions are
mirrored. Optionally drop the `data/config.json.embedding*` fields
to confirm the daemon's "RAG disabled" fallback.

If anything went wrong, share the `/data/logs/{run,daemon,web}.log`
files and the output of `pnpm scripts/lint` — those three cover 95%
of v0.3 issues.
